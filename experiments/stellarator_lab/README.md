# Stellarator Lab

Discrete-exterior-calculus machinery on the 600-cell, tagged at
`stellarator-lab-foundations`. The math is real and well-tested
(machine-precision verifications); the predictive utility on real
ML benchmarks is not established by this work alone.

## What's here

| File | Purpose |
|---|---|
| `hopf_seed.py` | Closed-form generalized-Hopf seed field B(x; ω₁, ω₂) on R³, with closed-form magnetic helicity H = 2π² · nm · n!m!/(n+m)! · R⁴ verified empirically across 18 (n,m) pairs to 1e-12 |
| `hopf_grid.py` | Cartesian grid sampling, central-difference div / curl / Simpson helicity, boundary-flux check |
| `hopf_600cell_witness.py` | Irrep-native Hodge decomposition on the 600-cell with **machine-zero** harmonic leak (~6×10⁻¹⁶), verifying the load-bearing math |

## Key facts (the load-bearing math)

- **2I acts freely on the 720 edges of the 600-cell** → the edge
  representation is exactly six copies of the regular rep:
  `6·𝟙 ⊕ 12·𝟚_a ⊕ 12·𝟚_b ⊕ 18·𝟛_a ⊕ 18·𝟛_b ⊕ 24·𝟜_a ⊕ 24·𝟜_b ⊕ 30·𝟝 ⊕ 36·𝟞`
  (proved by character argument: only the antipodal pair could
  contribute, but antipodal vertices are at angular distance π,
  not the 600-cell edge distance π/5).
- **Hodge stars are scalar × identity per cochain space** because
  the 600-cell is regular under H₄ (every k-cell is in a single
  H₄-orbit). 2I-equivariance is automatic.
- **Closed-form helicity** H(n, m, R) = 2π² · nm · n!m!/(n+m)! · R⁴.
  Empirically verified to machine precision; the derivation
  (DLMF §5.12 + §14.30 + §34.3) is sketched in this directory's git
  history but not written out as a standalone proof.
- The **600-cell discrete Hopf decomposition** partitions 120
  vertices into 12 great-circle decagons of 10 vertices each, as
  left cosets of a C₁₀ subgroup of 2I generated by the icosian
  q = (φ/2, 1/2, 1/(2φ), 0). This is what `snake_hopf_controller.py`
  in the parent directory uses for its action geometry.

## Where this connects to the GENREG runtime

The `hopf_decagon.py` module at the top level (mirror copy) is
imported by `snake_hopf_controller.py`, which exposes a
`SnakeHopfController` that GENREG `Genome` can use as a controller
backend (`controller_type="snake_hopf"`). See `bench_snake.py` in
the repo root for the Snake A/B benchmark of MLP vs HopfController
vs SnakeHopfController.

## What this is NOT

This is not a stellarator equilibrium solver and not a contribution
to the QS optimization pipeline. The original research-program
framing (six layers from Hopf seed through equilibrium + Boozer
optimization) was scoped down after a market check (Q3 of the
former CONSULTATION.md): the Landreman QUASR catalogue uses
Garren–Boozer near-axis expansion + VMEC/DESC for QS; the Hopfion
family doesn't compete with that pipeline directly. The math here
remains a clean methods package; whether it has a future
ML/physics application beyond the GENREG-controller use is open.
